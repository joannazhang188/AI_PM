# AI 工作流总说明

## 1. 目的

本说明用于统一产品经理工作台中“大模型回复逻辑”的设计方式，保证：

- 快捷功能对应明确的工作流，而不是一段松散提示词
- 前端展示状态与后端实际执行阶段一致
- 文档生成、编辑、引用、导出、资源库存储使用同一份真实内容
- 后续继续细化 prompt 或替换模型时，不需要再推翻整体结构

本阶段不定义具体 prompt 文案，只定义工作流边界、输入输出协议和前后端职责。

## 2. 总原则

### 2.1 快捷功能 = 工作流

快捷功能不是 UI 标签，也不是单纯的支线提示词，而是一个完整工作流定义，至少包含：

- 工作流目标
- 输入要求
- 是否需要追问澄清
- 是否需要先确认模板
- 输出物类型
- 是否支持基于原文修改
- 是否需要联网

### 2.2 任一时刻最多只有一个有效工作流

- 用户本轮消息只能使用 0 或 1 个快捷功能
- 当用户切回“选择快捷功能”时，表示本轮不使用任何工作流
- 本轮发送时，以输入区当前选中的工作流为准
- 不允许同时套用多个工作流

### 2.3 文档必须有单一真源

无论文档来自：

- 模型生成
- 用户编辑
- 资源库引用
- 历史会话引用

都必须落到统一的 `artifact.content` 上。

后续的：

- 右侧预览
- 再次引用
- 继续修改
- 导出本地
- 导出到资源库

都必须基于最新 `artifact.content`。

### 2.4 资源库保存的必须是真实内容

资源库中的资源不是一个壳引用，而是包含：

- 正文内容
- 格式信息
- 来源工作流
- 来源会话
- 创建和更新时间

再次引用资源库时，进入模型上下文的必须是资源正文，而不是资源名。

### 2.5 后端必须返回结构化阶段信息

前端不能再靠猜测判断“当前回复是在追问、确认模板还是生成文档”。  
后端响应必须明确返回当前阶段。

## 3. 统一工作流阶段

所有快捷功能统一使用以下阶段机。

### 3.1 `materials_intake`

含义：

- 接收用户文本
- 读取上传文件
- 读取上传图片
- 读取引用文档
- 读取引用资源库内容

目标：

- 判断素材是否足够
- 提取可用上下文

### 3.2 `clarify`

适用场景：

- 素材不完整
- 关键约束缺失
- 输出结果强依赖业务细节

目标：

- 向用户提出关键问题
- 通过多轮对话补齐信息

要求：

- 问题要围绕最终产物的关键信息
- 优先问少量高价值问题，不要一次性罗列过多问题

### 3.3 `template_confirm`

适用场景：

- 需要输出正式文档、报告、HTML 原型、PPT 内容等成品

目标：

- 先让用户确认模板、大纲或输出结构

模板来源优先级：

1. 用户上传模板
2. 用户引用模板文档
3. 用户明确描述的模板要求
4. 系统默认模板
5. 模型建议模板

规则：

- 若用户已明确说“按默认模板直接生成”，可视为已确认
- 若用户上传了模板且表示直接按此生成，可跳过模板提议阶段

### 3.4 `draft_generate`

目标：

- 根据已确认模板和素材生成初稿

输出形式：

- `md`
- `html`
- `json`
- `code`
- 表格型文档

此阶段应显式返回 `artifacts`。

### 3.5 `revise_existing`

适用场景：

- 用户引用已有文档并要求修改
- 用户明确指出“基于当前版本修改”

规则：

- 必须基于引用文档的内容进行局部或整体修改
- 不能忽略原文直接重写一份无关的新文档
- 返回结果应是更新后的完整文档版本

### 3.6 `finalize`

目标：

- 输出最终成品
- 支持导出本地
- 支持导出资源库

## 4. 用户通用行为规则

### 4.1 生成前先确认模板

对所有“成品型”工作流，生成前必须先确认模板或默认大纲。

成品型工作流包括：

- 需求澄清摘要
- PRD
- PRD 评审
- 生成原型图
- Story
- 测试用例
- 测试用例评审
- 产品功能更新说明
- 产品竞品分析
- 产品调研报告
- 产品介绍 PPT

### 4.2 修改时默认基于原文

只要用户：

- 引用了已有文档
- 或明确指定要修改某份文档

后端就必须进入 `revise_existing` 阶段。

### 4.3 未选择快捷功能时走普通助手模式

若本轮 `workflowId = null`，则：

- 模型只理解当前输入和引用素材
- 不进入任何专用工作流
- 不强制提问、确认模板或产出特定文档

## 5. 输出物统一模型

建议统一使用 `artifact` 作为所有正式产物的抽象。

```ts
type Artifact = {
  id: string;
  conversationId: string;
  workflowId: string | null;
  type: "document" | "html" | "code" | "table";
  format: "md" | "html" | "json" | "text";
  title: string;
  content: string;
  createdAt: string;
  updatedAt: string;
  version: number;
  isEdited: boolean;
};
```

说明：

- `content` 是唯一真源
- `preview` 一律由 `content` 派生
- 历史消息和资源库只存 `artifactId` 或资源快照，不再复制多份业务正文

## 6. 资源库统一模型

```ts
type Resource = {
  id: string;
  sourceArtifactId: string;
  workflowId: string | null;
  workflowName: string;
  name: string;
  format: string;
  content: string;
  conversationId: string;
  conversationTitle: string;
  createdAt: string;
  updatedAt: string;
  isEdited: boolean;
};
```

## 7. 推荐实施顺序

### 第一阶段：统一协议和阶段机

- 建立 workflow registry
- 明确 chat request / response 协议
- 后端响应返回 `stage`
- 前端按阶段渲染

### 第二阶段：统一 artifact / resource 真源

- 文档编辑后回写 artifact
- 引用统一走 artifactId / resourceId
- 导出资源库保存正文

### 第三阶段：细化 11 个工作流

- 模板规则
- 追问策略
- 联网搜索策略
- 输出结构细节
