# 快捷功能工作流定义

## 1. 使用说明

每个快捷功能都按同一结构定义：

- 目标
- 输入
- 是否需要先澄清
- 是否必须先确认模板
- 输出物
- 修改规则
- 联网要求

## 2. 工作流定义

## 2.1 需求澄清

- `workflowId`: `requirement`
- 目标：通过多轮对话沟通、求证、补充和澄清需求，最终形成需求理解摘要
- 允许输入：文本、文件、图片、引用资源、引用文档
- 默认阶段：`materials_intake -> clarify -> template_confirm -> draft_generate`
- 关键行为：
  - 重点追问影响产品方案的关键细节
  - 可穿插调研建议和补充问题
  - 不应过早直接输出总结
- 输出物：`md` 需求理解摘要
- 修改规则：引用需求摘要后，按原文修改
- 联网：默认不强制

## 2.2 生成 PRD

- `workflowId`: `prd`
- 目标：基于需求摘要或需求素材生成 PRD 文档
- 允许输入：文本、文件、图片、引用需求摘要、引用资源
- 默认阶段：`materials_intake -> template_confirm -> draft_generate`
- 关键行为：
  - 先确认 PRD 模板或大纲
  - 再输出正式 PRD
- 输出物：`md` PRD 文档
- 修改规则：引用 PRD 后，按原文修改
- 联网：不强制

## 2.3 PRD 评审

- `workflowId`: `prdReview`
- 目标：从产品、技术、业务角度对 PRD 进行评审
- 允许输入：PRD、需求摘要、文本补充说明
- 默认阶段：`materials_intake -> template_confirm -> draft_generate`
- 关键行为：
  - 先确认评审维度或评审模板
  - 再输出评审意见
- 输出物：`md` 评审报告
- 修改规则：引用评审报告后可继续修改和补充
- 联网：不强制

## 2.4 生成原型图

- `workflowId`: `prototype`
- 目标：根据 PRD 或需求摘要生成可预览原型图
- 允许输入：PRD、需求摘要、图片、文本
- 默认阶段：`materials_intake -> template_confirm -> draft_generate`
- 关键行为：
  - 先确认原型页面范围和信息结构
  - 再输出可预览 HTML
- 输出物：`html` 文件
- 修改规则：引用原型 artifact 后按原 HTML 修改
- 联网：不强制

## 2.5 Story

- `workflowId`: `story`
- 目标：根据 PRD 或需求描述生成 Story 任务
- 允许输入：PRD、需求摘要、文本
- 默认阶段：`materials_intake -> template_confirm -> draft_generate`
- 关键行为：
  - 先确认 Story 模板和拆分粒度
  - 再输出 Story 列表
- 输出物：`md` Story 文档
- 修改规则：引用 Story 后按原文修改
- 联网：不强制

## 2.6 测试用例

- `workflowId`: `testCase`
- 目标：根据 Story 生成测试用例
- 允许输入：Story、PRD、文本补充说明
- 默认阶段：`materials_intake -> template_confirm -> draft_generate`
- 关键行为：
  - 先确认测试用例模板和覆盖粒度
  - 再输出测试用例
- 输出物：`md` 或表格型文档
- 修改规则：引用测试用例后按原文修改
- 联网：不强制

## 2.7 测试用例评审

- `workflowId`: `testReview`
- 目标：根据 PRD 评估测试用例是否覆盖完整场景
- 允许输入：PRD、测试用例、文本补充
- 默认阶段：`materials_intake -> template_confirm -> draft_generate`
- 关键行为：
  - 先确认评审维度
  - 再输出缺失场景、风险点和改进建议
- 输出物：`md` 评审报告
- 修改规则：引用评审报告后按原文补充或修订
- 联网：不强制

## 2.8 产品功能更新说明

- `workflowId`: `releaseNote`
- 目标：根据 PRD 生成本次产品功能更新说明
- 允许输入：PRD、需求摘要、文本
- 默认阶段：`materials_intake -> template_confirm -> draft_generate`
- 关键行为：
  - 先确认更新说明模板
  - 再输出正式说明
- 输出物：`md` 更新说明
- 修改规则：引用更新说明后按原文修改
- 联网：不强制

## 2.9 产品竞品分析

- `workflowId`: `competitorAnalysis`
- 目标：根据输入需求联网搜索相关竞品，输出竞品分析报告
- 允许输入：文本、文件、图片、引用资源
- 默认阶段：`materials_intake -> clarify -> template_confirm -> draft_generate`
- 关键行为：
  - 若目标市场或竞品范围不明确，应先澄清
  - 必须联网搜索
  - 输出需带来源信息
- 输出物：`md` 竞品分析报告
- 修改规则：引用报告后按原文修订
- 联网：必须

## 2.10 产品调研报告

- `workflowId`: `productResearch`
- 目标：根据输入需求联网搜索相关内容，输出产品调研报告
- 允许输入：文本、文件、图片、引用资源
- 默认阶段：`materials_intake -> clarify -> template_confirm -> draft_generate`
- 关键行为：
  - 若调研方向不明确，应先澄清
  - 必须联网搜索
  - 输出需带来源信息
- 输出物：`md` 调研报告
- 修改规则：引用报告后按原文修订
- 联网：必须

## 2.11 产品介绍 PPT

- `workflowId`: `productPresentation`
- 目标：根据需求文档内容输出产品介绍
- 允许输入：需求摘要、PRD、更新说明、文本
- 默认阶段：`materials_intake -> template_confirm -> draft_generate`
- 关键行为：
  - 先确认 PPT 页面结构或内容模板
  - 第一阶段建议输出逐页内容的 `md`
- 输出物：`md` 版 PPT 大纲或逐页文案
- 修改规则：引用后按原文逐页修改
- 联网：不强制

## 3. 所有工作流的共性规则

### 3.1 成品生成前先确认模板

若用户未提供模板，则模型需先输出默认大纲或模板示例。  
只有用户确认后，才能生成正式成品。

### 3.2 用户上传模板优先

模板优先级如下：

1. 用户上传模板
2. 用户引用模板
3. 用户明确描述的模板要求
4. 系统默认模板

### 3.3 用户要求修改时基于原文修改

当用户：

- 引用了已有 artifact
- 或引用了资源库中已有正式内容
- 或明确指定“按刚才文档修改”

后端必须进入 `revise_existing`。

### 3.4 资源库内容必须可被模型理解

资源库再次引用时，进入模型上下文的必须是资源正文，而不是资源标题。
