# 产品经理工作台落地方案

## 1. 结论先行

这套需求最适合用 **“1 个 AI 应用 + 7 条对话流 + 插件/HTTP 节点直连能力 + 飞书沉淀”** 的方式落地。

- 第一阶段先做 7 条支线，不做原型线。
- 每条支线独立，不自动串联。
- 每条支线都用 **对话流** 承载，而不是普通工作流。
- 最终保存不自动发生，统一通过用户在对话中显式触发“保存当前版本”完成。
- 文档型结果沉淀到 **飞书文档**，结构化结果沉淀到 **飞书多维表格**。
- 扣子知识库只承担“模板、规范、示例、方法论”的辅助检索，不承担正式结果存档。

## 2. 为什么这样设计

结合需求说明和扣子文档，核心原因有 4 个：

1. 你的应用本质是“在某条支线内持续对话和修改”，而不是一次性任务执行，所以主载体必须是 **对话流**。
2. 7 条支线之间明确要求独立、人工衔接，因此不应设计成一条自动串联大工作流。
3. 只有最终确认版本才保存，因此保存动作不能挂在正常生成链路上自动执行，必须单独做成“保存分支”。
4. Story 和测试用例要求标准字段，最稳妥的方式不是让模型自由输出长文本，而是让模型输出 **JSON 结构化结果**，再写入多维表格。

## 3. 总体架构

```text
产品经理工作台（AI 应用）
├── 支线1：需求线对话流
├── 支线2：PRD 线对话流
├── 支线3：PRD 评审线对话流
├── 支线5：Story 线对话流
├── 支线6：测试用例线对话流
├── 支线7：测试用例评审线对话流
├── 支线8：产品功能更新说明线对话流
├── 插件/HTTP：文件解析
├── 模型能力：图片理解/补充描述
├── 插件/HTTP：保存到飞书文档
├── 插件/HTTP：保存到飞书多维表格
└── 知识资源：模板库 / 规范库 / 示例库
```

## 4. 产品层搭建方式

### 4.1 容器层

先创建 1 个 AI 应用，作为“产品经理工作台”的总入口。

应用首页只放 7 个入口：

- 需求线
- PRD 线
- PRD 评审线
- Story 线
- 测试用例线
- 测试用例评审线
- 产品功能更新说明线

原型线先保留在产品结构里，但第一阶段不上线流程。

### 4.2 每条支线的实现单元

每条支线各自创建 1 条 **对话流**。

每条对话流内部都遵循同一套骨架：

1. 开始节点
2. 意图识别节点
3. 资料预处理节点
4. 主生成节点
5. 结果整理节点
6. 保存判断分支
7. 结束节点

## 5. 单条支线的标准流程

建议每条支线统一采用下面这套编排。

### 5.1 对话入口

开始节点接收：

- `USER_INPUT`
- `CONVERSATION_NAME`

同时允许用户上传：

- 文本
- 文件
- 图片

### 5.2 输入预处理

用户输入可能是混合内容，所以每条支线都加一个“输入预处理层”：

- 文本直接进入主模型
- 文件通过“链接读取/文件解析”能力转成纯文本
- 图片通过视觉理解模型转成图片描述或从图中抽取文字
- 代码节点把文本、文件解析结果、图片解析结果合并成统一上下文

统一沉淀为一个中间变量：

- `source_context`

这样主模型只消费一个统一输入，不会在每条支线里反复拼接逻辑。

### 5.3 意图识别

每条对话流都先做一层意图识别，至少分成 4 类：

- 生成初稿
- 修改当前结果
- 解释或补充某一段内容
- 保存当前版本

这样用户说“把这段写得更偏业务”“删掉风险部分”“保存当前版本”，都能在同一条支线内处理，而不需要切换流程。

### 5.4 主生成链路

每条支线的主生成节点使用大模型节点，配固定模板提示词。

建议所有支线都采用：

- 系统提示词：角色、边界、输出模板、禁忌规则
- 用户提示词：本轮用户输入 + 已有结果 + source_context
- 开启对话历史
- 对需要长文输出的支线开启续写

### 5.5 当前版本管理

不建议把正式结果直接依赖“历史消息里最后一条”。

更稳妥的方式是：在每条支线里配置应用变量或支线变量，至少维护以下字段：

- `current_draft`
- `current_structured_result`
- `latest_summary`
- `last_saved_version_id`

每次主模型生成新结果后，用变量赋值节点更新 `current_draft` 或 `current_structured_result`。

这样用户后续说“基于刚才那版继续改”，系统取的是变量中的当前草稿，而不是脆弱地回捞整段会话。

建议输出模板：

- 背景

- 用户问题

## 6.2 PRD 线

目标：基于需求说明持续生成和修改 PRD。

建议输出模板：

- 产品背景
- 目标与范围
- 用户角色
- 业务流程
- 功能清单
- 异常与边界
- 数据与埋点要求
- 非功能要求
- 验收标准

保存介质：飞书文档

实现方式：

- 主模型输出 Markdown
- 允许多轮追改
- 保存时写入飞书文档

## 6.3 PRD 评审线

目标：从产品、技术、业务三个视角输出评审意见。

建议输出模板：

- 产品视角问题
- 技术视角风险
- 业务视角风险
- 优先级建议
- 必改项
- 可优化项

保存介质：飞书文档

## 6.4 Story 线

目标：把 PRD 转成标准字段的 Story。

不要直接输出自由文本，必须输出 JSON 数组。

建议字段：

- `story_id`
- `story_title`
- `story_description`
- `role`
- `goal`
- `benefit`
- `acceptance_criteria`
- `priority`
- `related_module`
- `dependency`
- `source_prd`

保存介质：飞书多维表格

实现方式：

- 大模型节点输出 JSON
- 代码节点把 JSON 映射成飞书多维表格 `records` 数组
- 插件节点或 HTTP/飞书接口写入表格

## 6.5 测试用例线

目标：根据 Story 生成标准化测试用例。

建议字段：

- `case_id`
- `case_title`
- `precondition`
- `test_steps`
- `expected_result`
- `case_type`
- `priority`
- `related_story_id`
- `remark`

保存介质：飞书多维表格

实现方式同 Story 线。

## 6.6 测试用例评审线

目标：根据 PRD 评估测试用例覆盖是否完整。

建议输出模板：

- 已覆盖场景
- 未覆盖场景
- 高风险遗漏
- 建议补充用例
- 结论

保存介质：飞书文档

## 6.7 产品功能更新说明线

目标：根据 PRD 输出面向内外部的版本更新说明。

建议输出模板：

- 更新背景
- 新增能力
- 优化内容
- 使用影响
- 注意事项

保存介质：飞书文档

## 7. 输入能力怎么落地

### 7.1 文本输入

直接用对话流原生输入。

### 7.2 文件输入

文件本身不能直接被模型读取，建议直接通过插件节点或大模型挂载工具能力完成解析：

1. 接收 File 类型输入
2. 调用链接读取或文件解析插件
3. 输出纯文本
4. 将解析结果直接传给当前支线主流程

支持的典型文件：

- PDF
- Word
- Excel

### 7.3 图片输入

图片不要当普通输入塞给文本参数，直接走视觉理解输入。

建议两类能力：

- 图片转文字
- 图片内容摘要

适用场景：

- 需求草图截图
- 原型截图
- 业务流程图截图
- 表格截图

## 8. 保存机制怎么落地

这是整套方案里最关键的部分。

### 8.1 保存原则

- 不自动保存过程稿
- 只保存用户确认版本
- 保存动作必须由用户显式触发

### 8.2 触发方式

最稳妥的方式不是前端按钮，而是对话内指令：

- “保存当前版本”
- “把这版存为 PRD”
- “确认保存”

由意图识别节点命中“保存当前版本”后，进入保存分支。

### 8.3 文档型结果保存

适用于：

- 需求说明
- PRD
- PRD 评审
- 测试用例评审
- 更新说明

实现建议：

1. 当前对话流拿到 `current_draft`
2. 代码节点做标题、目录、时间戳、版本号拼装
3. 通过飞书插件或 HTTP 节点直接写入飞书文档
4. 成功后回写 `last_saved_version_id`

### 8.4 结构化结果保存

适用于：

- Story
- 测试用例

实现建议：

1. 大模型输出 JSON
2. 代码节点转为飞书多维表格要求的 `records` 数组
3. 通过飞书多维表格插件或 HTTP 节点直接写入
4. 返回写入成功条数

## 9. 飞书落地建议

结合现有文档，飞书这块建议分两套：

### 9.1 飞书文档

用于长文本正式稿。

建议按支线分目录：

- `/产品经理工作台/需求说明/`
- `/产品经理工作台/PRD/`
- `/产品经理工作台/PRD评审/`
- `/产品经理工作台/测试评审/`
- `/产品经理工作台/更新说明/`

文档命名建议：

- `项目名_支线名_版本号_日期`

例如：

- `增长中台_PRD_V1_2026-03-22`

### 9.2 飞书多维表格

用于 Story、测试用例等结构化结果。

建议至少建 2 张表：

- `Story`
- `TestCases`

并补充公共字段：

- `project_name`
- `source_doc`
- `version`
- `created_by`
- `created_at`
- `status`

## 10. 知识库和数据库怎么用

### 10.1 知识库

建议只存：

- 各支线输出模板
- PRD 模板
- Story 字段规范
- 测试用例字段规范
- 优秀示例
- 产品方法论资料

不要把正式产出依赖知识库存档。

### 10.2 数据库

如果你只是做第一阶段 MVP，其实可以先不用数据库。

只有在你后面要支持以下能力时，再引入数据库：

- 保存草稿索引
- 查询历史保存记录
- 做版本列表页
- 记录项目级元数据

第一阶段优先把正式结果直接沉淀到飞书，复杂版本管理先不做，和你的需求完全一致。

## 11. 提示词体系怎么搭

每条支线都不要把提示词写成一段散文，应该做成固定模板。

每条支线都拆成三层：

1. 角色层：你是谁
2. 规则层：你必须遵守什么
3. 输出层：你必须按什么格式输出

建议把这些提示词沉淀成资源库里的独立提示词，按支线复用。

例如：

- `prompt_requirement_clarifier`
- `prompt_prd_writer`
- `prompt_prd_reviewer`
- `prompt_story_writer`
- `prompt_testcase_writer`

## 12. 第一阶段最优上线范围

不要 7 条一起做。

第一阶段建议先上线 4 条：

1. 需求线
2. PRD 线
3. Story 线
4. 测试用例线

原因：

- 这是从模糊需求到执行拆解的主价值链
- 文档型和结构化两种保存方式都能覆盖
- 能最快验证“多轮修改 + 人工确认保存”是否顺手

第二阶段再补：

- PRD 评审线
- 测试用例评审线
- 产品功能更新说明线

## 13. 你现在还必须先补齐的 3 个定义

在真正搭建前，最需要先定死的是这 3 件事：

1. Story 字段规范
2. 测试用例字段规范
3. 7 条支线的最终输出模板

如果这三件事不先固定，后面的模型提示词、JSON 输出、飞书字段映射都会反复返工。

## 14. 实施顺序

建议按下面顺序搭建：

1. 先定义 7 条支线模板
2. 定义 Story 和测试用例字段
3. 建飞书文档目录和多维表格
4. 创建模板知识库
5. 接好文件解析插件能力
6. 接好飞书文档写入能力
7. 接好飞书多维表格写入能力
8. 先做需求线对话流
9. 复制骨架做 PRD / Story / 测试用例线
10. 联调保存分支
11. 做评审类支线

## 15. 一句话定版

最终落地方案不是“一个大流程自动跑完产品全链路”，而是：

**用 1 个产品经理工作台应用承载 7 条独立对话流，每条支线支持混合输入、多轮修改、人工确认保存，并通过飞书文档/飞书多维表格完成正式结果沉淀。**
