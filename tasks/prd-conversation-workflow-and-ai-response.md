# PRD: 对话工作流与 AI 回复机制

## 1. 背景

当前产品经理工作台的 UI 结构已经基本完成，下一阶段需要明确对话过程中 AI 如何围绕“快捷功能”工作，如何处理多轮修改、引用文档、导出资源、以及长文档在会话中的展示方式。

本说明面向前后端开发，目标是统一以下能力：

- 对话绑定快捷功能
- 多输入源统一进入模型上下文
- 多轮对话下的增量修改逻辑
- 长文档生成与卡片化展示
- 文档引用与资源库协作

## 2. 目标

1. 用户在进入对话后，可以明确选择一个快捷功能，并让后续 AI 回复始终围绕该功能展开。
2. 文本、文件、图片、资源库引用内容能够统一作为当前轮的输入上下文。
3. 用户在同一快捷功能下继续修改结果时，AI 基于已有结果增量修改，而不是默认重写。
4. 长文档型输出能够在会话中以文档卡片形式展示，并可进一步引用、查看详情、导出和沉淀到资源库。
5. 前后端之间对会话、文档、资源、上下文组装方式有一致的数据协议。

## 3. 核心使用路径

### 3.1 标准路径

1. 用户进入对话。
2. 选择某个快捷功能。
3. 通过文本、文件、图片输入信息。
4. AI 围绕该快捷功能回复用户，完成当前任务。
5. 用户如不满意，可以继续修改。
6. 如生成了长文档，可卡片化展示，并支持查看详情、引用和导出。

### 3.2 示例路径

#### 示例 A：生成 PRD

1. 用户选择 `生成PRD`
2. 输入文本、上传文件、上传图片
3. AI 识别这些输入为 PRD 素材
4. AI 输出 PRD 文档
5. 前端将 PRD 以文档卡片方式插入会话

#### 示例 B：需求评审

1. 用户切换为 `需求评审`
2. 引用刚刚生成的 PRD 文档
3. 再补充文本说明本次评审关注点
4. AI 对“引用 PRD + 当前文本输入”做评审
5. 输出评审结果

#### 示例 C：原文档增量修改

1. 用户仍停留在 `生成PRD`
2. 引用当前 PRD 文档
3. 输入“修改某一处”
4. AI 在当前 PRD 基础上修改，而不是重新生成一份全新 PRD

## 4. 关键业务规则

### 4.1 快捷功能绑定规则

1. 每个会话在任一时刻有一个“当前快捷功能”。
2. AI 的回复必须优先服从当前快捷功能的任务目标。
3. 若用户切换快捷功能，则后续轮次的系统提示词、任务类型、输出格式要求随之切换。
4. 历史消息仍保留，但“当前快捷功能”优先决定本轮任务解释方式。

### 4.2 输入上下文合并规则

每一轮提交给后端/模型的上下文由以下几类组成：

1. 当前快捷功能
2. 当前轮文本输入
3. 当前轮上传文件
4. 当前轮上传图片
5. 当前轮引用的资源库内容
6. 当前会话历史
7. 当前会话中被引用的文档/代码卡片

说明：

- 前端负责明确“本轮引用了哪些内容”
- 后端负责把这些输入组装成模型可理解的统一上下文

### 4.3 多轮修改规则

1. 若用户在同一快捷功能下继续修改已有结果，且本轮引用了旧文档，则 AI 必须将该文档视为“待修改对象”。
2. 模型不可默认重写整份文档，除非用户明确要求“重写”。
3. 若用户未引用旧文档，则模型可按当前输入重新生成结果。
4. 后端需要在请求中显式告诉模型：
   - 当前快捷功能
   - 是否存在“待修改文档”
   - 待修改文档内容
   - 用户本轮修改指令

### 4.4 长文档输出规则

当 AI 输出为较长结构化文档时，不应仅以纯聊天文本渲染。

应满足：

1. 会话中展示摘要或说明文字
2. 同时插入文档卡片
3. 文档卡片至少包含：
   - 文档名称
   - 文档类型/格式
   - 引用按钮
4. 点击卡片可打开右侧详情面板
5. 详情面板支持：
   - 预览
   - 代码/原文
   - 导出
   - 导出到资源库

### 4.5 引用规则

1. 点击会话中的文档卡片“引用”，应将该文档加入输入框下方的引用区。
2. 引用展示方式与上传文件/图片保持一致。
3. 本轮发送时，该引用内容必须进入模型上下文。
4. 引用可取消。

## 5. 前端需求

## 5.1 会话输入区

前端需支持以下输入源同时存在：

- 文本
- 文件
- 图片
- 语音转文字
- 快捷功能标签
- 资源引用标签
- 文档引用标签

发送时前端需向后端提交明确结构：

```ts
type SendPayload = {
  conversationId: string | null;
  activeFunction: string;
  text: string;
  files: UploadedFileRef[];
  images: UploadedImageRef[];
  citedResources: ResourceRef[];
  citedDocuments: DocumentRef[];
};
```

## 5.2 会话消息渲染

前端需支持以下消息形态：

1. 用户文本消息
2. AI 普通文本回复
3. AI 长文档回复
4. 文档卡片
5. 资源卡片
6. 代码片段

其中长文档回复推荐拆成两层：

- 聊天区摘要文本
- 独立文档卡片

## 5.3 文档面板

前端在右侧详情面板需展示：

- 文档标题
- 文档生成时间
- 文档格式
- 预览内容
- 原始内容
- 导出菜单

导出菜单项：

- 导出为本地 pdf
- 导出为本地 md
- 导出为本地 doc
- 导出到资源库

## 5.4 资源库页

资源库页需要展示资源表格：

- 资源 ID
- 资源名称
- 所属类型
- 创建日期
- 说明
- 操作

无记录时显示：

`暂无保存资源`

点击“引用至对话”时：

- 若当前存在会话，则回到当前会话
- 将资源作为引用标签插入输入区

## 6. 后端需求

## 6.1 会话请求编排

后端需要将前端提交内容整理成模型请求：

```ts
type ModelContext = {
  activeFunction: string;
  functionInstruction: string;
  conversationHistory: Message[];
  currentText: string;
  uploadedFiles: ParsedFileContent[];
  uploadedImages: ParsedImageContent[];
  citedResources: ResourceContent[];
  citedDocuments: DocumentContent[];
  targetDocumentForEdit?: DocumentContent | null;
};
```

### 6.2 模型指令生成

后端需根据 `activeFunction` 注入不同系统提示词。

例如：

- `生成PRD`
  - 把所有输入视为 PRD 素材
  - 输出应符合 PRD 模板
- `需求评审`
  - 把输入视为待评审对象
  - 输出为评审意见
- `生成测试用例`
  - 输出结构化测试用例

### 6.3 增量修改判断

后端需识别本轮是否属于“修改已有文档”：

判断条件建议：

1. 当前快捷功能未变
2. 用户引用了会话内已有文档或资源库文档
3. 用户文本中表达修改、补充、调整、优化等意图

若命中，则系统提示词中要明确：

- 基于已有文档修改
- 不要默认重写
- 非必要不改变未提及部分

### 6.4 长文档输出结构

后端返回前端时，不建议只返回一整段纯文本。

建议返回结构化结果：

```ts
type AssistantReply = {
  textReply: string;
  artifacts: Artifact[];
};

type Artifact = {
  id: string;
  kind: "document" | "code" | "table";
  title: string;
  format: "md" | "json" | "html" | "txt";
  content: string;
  summary?: string;
  createdAt: string;
  functionType: string;
};
```

其中：

- `textReply` 用于聊天区说明
- `artifacts` 用于卡片化展示

### 6.5 资源库存储

后端需提供资源库存储能力，保存以下字段：

```ts
type ResourceItem = {
  id: string;
  name: string;
  type: string;
  createdAt: string;
  description: string;
  format: string;
  content: string;
  sourceConversationId: string;
  sourceArtifactId?: string;
};
```

保存规则：

1. 导出到资源库时生成唯一资源 ID
2. 资源名称取当时输出的文档名称
3. 所属类型取当前快捷功能
4. 创建日期取文档生成日期
5. 说明取当前对话标题
6. 内容按原格式保存，如 `md`、`json`

## 7. 推荐接口

### 7.1 发送消息

`POST /api/conversations/:conversationId/messages`

请求体：

```json
{
  "activeFunction": "生成PRD",
  "text": "请基于以下内容生成PRD",
  "files": [],
  "images": [],
  "citedResources": [],
  "citedDocuments": []
}
```

响应体：

```json
{
  "messageId": "msg_xxx",
  "textReply": "我已基于你提供的资料生成 PRD。",
  "artifacts": [
    {
      "id": "doc_xxx",
      "kind": "document",
      "title": "狼人杀产品需求文档",
      "format": "md",
      "content": "# PRD ...",
      "createdAt": "2026-03-23 20:45",
      "functionType": "生成PRD"
    }
  ]
}
```

### 7.2 导出到资源库

`POST /api/resources`

### 7.3 查询资源库

`GET /api/resources`

### 7.4 删除资源

`DELETE /api/resources/:resourceId`

## 8. 非目标

当前阶段不做：

1. 多快捷功能自动串联工作流
2. 不引用文档时的复杂意图识别推理链
3. 全量版本管理与文档 diff 对比
4. 资源库全文检索
5. 权限、协同、审批流

## 9. 验收标准

1. 用户选择快捷功能后，AI 后续回复与当前功能一致。
2. 文本、文件、图片、引用文档都能进入同一轮模型上下文。
3. 同一快捷功能下引用旧文档继续修改时，AI 输出为增量修改结果，而不是默认重写。
4. 长文档输出时，会话中出现文档卡片。
5. 文档卡片可打开详情面板，可导出到资源库。
6. 资源库能显示保存记录，空状态能显示“暂无保存资源”。
7. 点击“引用至对话”可将资源回填至输入区。

## 10. 开放问题

1. 是否需要对“修改已有文档”增加显式模式开关，而不是完全依赖引用判断？
2. 测试用例、需求评审等结构化输出，是否需要定义统一 artifact schema？
3. 资源库存储是否只保存最终确认版本，还是允许保存中间版本？
4. 是否需要对不同快捷功能分别定义固定模板与校验器？
