# PRD: 扣子需求澄清对话流

## 1. 概述

搭建一个运行在扣子平台中的对话流，面向“需求还很模糊、只有一个初步想法”的用户。系统接收用户输入的文字、图片、文件，先做内容识别与解析，再判断当前对话应该进入“继续澄清”“生成需求理解摘要”“改写摘要”还是“保存摘要”分支，最终输出结构化的需求理解摘要，并支持将最近一次摘要写入数据库。

该对话流重点不是追问到底，而是以较轻量的 3-5 个问题帮助用户把目标、对象、范围、约束、交付结果补齐到“大致可理解、可继续推进”的程度。

## 2. 目标

- 支持用户以文字、图片、文件中的任意一种或多种形式表达模糊需求。
- 自动识别是否存在图片或文件，并分流解析。
- 整合多模态输入，提取关键信息并判断下一步动作。
- 在需要澄清时，只提出 3-5 个高价值问题，避免盘问式交互。
- 在信息足够时，生成结构化需求理解摘要。
- 支持用户基于上一版摘要继续改写。
- 支持将最近一次摘要保存到扣子数据库。

## 3. 非目标

- 不在本期内生成完整 PRD、原型图、任务拆解或开发工单。
- 不做复杂的多轮表单式问卷。
- 不做数据库检索去重、版本回滚、权限审批。
- 不做 OCR/文件解析能力本身，解析依赖扣子可用插件或模型视觉理解能力。

## 4. 关键假设

- 使用“对话流”而不是普通工作流，因为需要读取会话历史并持续对话。
- 用户的图片可直接作为对话流输入传入，图片解析交给支持视觉理解的模型节点。
- 用户的文件解析使用插件节点，优先采用文档中提到的链接读取类插件。
- “最近一次摘要”通过数据库保存，必要时也可额外通过应用变量缓存。
- 对于步骤 7 的动作判断，采用意图识别节点比“模型+选择器”更稳。
- 摘要保存时，每次插入一条新记录，不在本期内处理更新历史版本。

## 5. 用户故事

### US-001: 输入模糊需求
**Description:** 作为用户，我希望直接发送一句模糊想法，或附带图片/文件，让系统理解我的需求背景。

**Acceptance Criteria:**
- [ ] 对话流开始节点可接收用户原始输入。
- [ ] 支持识别是否包含文本、图片、文件。
- [ ] 无图片和文件时，流程仍能正常继续。

### US-002: 自动解析多模态输入
**Description:** 作为用户，我希望系统自动读取我发来的图片和文件，而不是让我重复描述内容。

**Acceptance Criteria:**
- [ ] 图片输入可流转到图片解析分支。
- [ ] 文件输入可流转到文件解析分支。
- [ ] 图片/文件不存在时，对应分支不报错。
- [ ] 下游节点能统一拿到整合后的内容。

### US-003: 轻量澄清需求
**Description:** 作为用户，我希望系统只问必要问题，帮助我把需求说清楚。

**Acceptance Criteria:**
- [ ] 当关键信息缺失时，系统提出 1-5 个问题。
- [ ] 问题优先聚焦目标、用户对象、范围、约束、交付物。
- [ ] 问题形式可混合开放式、选项式、是否式。
- [ ] 若信息已足够，系统不再追问，直接生成摘要。

### US-004: 生成和改写需求摘要
**Description:** 作为用户，我希望系统能把我当前的需求理解整理成摘要，并按我的修改意见重写。

**Acceptance Criteria:**
- [ ] 可输出结构化需求理解摘要。
- [ ] 用户要求修改时，可基于最近一版摘要改写。
- [ ] 改写后的摘要覆盖旧摘要作为最近版本。

### US-005: 保存需求摘要
**Description:** 作为用户，我希望能把当前最新摘要保存到数据库，便于后续管理。

**Acceptance Criteria:**
- [ ] 仅保存最近一次生成或改写后的摘要。
- [ ] 保存成功后返回明确提示。
- [ ] 数据库至少保存会话名、原始输入、摘要正文、摘要版本时间。

## 6. 功能需求

- FR-1：对话流开始节点必须接收 `USER_INPUT` 和 `CONVERSATION_NAME`。
- FR-2：流程必须先判断输入中是否包含图片、文件、纯文字三类内容。
- FR-3：当包含图片时，流程必须进入图片解析分支。
- FR-4：当包含文件时，流程必须进入文件解析分支。
- FR-5：流程必须整合文字、图片解析结果、文件解析结果为统一语义上下文。
- FR-6：流程必须判断当前对话动作，动作至少包括：`clarify`、`generate_summary`、`rewrite_summary`、`save_summary`。
- FR-7：当动作是 `clarify` 时，流程必须判断当前是否仍存在待确认信息。
- FR-8：存在待确认信息时，流程必须生成 1-5 个问题；不存在时直接进入摘要生成。
- FR-9：当动作是 `generate_summary` 时，流程必须输出结构化需求理解摘要。
- FR-10：当动作是 `rewrite_summary` 时，流程必须基于最近一次摘要和用户新要求输出改写后的摘要。
- FR-11：当动作是 `save_summary` 时，流程必须将最近一次摘要写入数据库。
- FR-12：流程结束时必须返回自然语言结果给用户。

## 7. 对话流总体设计

### 7.1 推荐节点编排

1. 开始节点
2. 大模型节点：输入内容分析器
3. 选择器节点：是否有图片
4. 大模型节点：图片解析器
5. 选择器节点：是否有文件
6. 插件节点：文件解析器
7. 变量聚合节点：聚合图片解析结果
8. 变量聚合节点：聚合文件解析结果
9. 大模型节点：多模态整合与动作判断器
10. 意图识别节点：动作路由
11. 大模型节点：需求澄清问题生成器
12. 大模型节点：需求摘要生成器
13. 大模型节点：需求摘要改写器
14. 新增数据节点：保存需求摘要
15. 输出节点/结束节点

说明：
- 步骤 3 和步骤 5 原始需求里写成“并行”，在扣子里更稳妥的做法是分别做分支，再通过变量聚合统一收口。
- 如果平台当前支持更方便的并行子工作流，也可以把图片解析和文件解析封装成子工作流后并行调用。

### 7.2 12 步需求与节点映射

| 需求步骤 | 扣子节点实现 | 说明 |
| --- | --- | --- |
| 步骤1 开始 | 开始节点 | 读取 `USER_INPUT`、`CONVERSATION_NAME` |
| 步骤2 分析输入 | 大模型节点 | 判断是否含文字/图片/文件，输出信号 |
| 步骤3 判断信号 | 选择器节点 | 分流到图片/文件解析或跳过 |
| 步骤4 图片解析 | 大模型节点 | 视觉理解图片内容 |
| 步骤5 文件解析 | 插件节点 | 读取文件正文 |
| 步骤6 整合分析 | 大模型节点 | 汇总全部内容并判断动作 |
| 步骤7 判断动作 | 意图识别节点 | 路由到澄清/摘要/改写/保存 |
| 步骤8 需求澄清 | 大模型节点 | 输出 1-5 个问题或判定无需追问 |
| 步骤9 生成摘要 | 大模型节点 | 输出需求理解摘要 |
| 步骤10 改写摘要 | 大模型节点 | 基于最近摘要重写 |
| 步骤11 保存需求 | 新增数据节点 | 写入数据库 |
| 步骤12 结束 | 结束节点 | 输出最终结果 |

## 8. 节点详细配置

### 8.1 开始节点

对话流默认有：
- `USER_INPUT`：用户原始输入
- `CONVERSATION_NAME`：会话名称

建议补充自定义输入参数：
- `latest_summary`，String，非必填
  用于外部 API 调用时显式传入最近摘要；若不传，可由会话上下文承接。

### 8.2 大模型节点：输入内容分析器

**目的**
识别用户本轮输入中是否包含文字、图片、文件，并提取基础信号供后续分流。

**输入**
- `user_input` = `USER_INPUT`

**输出格式**
JSON：

```json
{
  "has_text": true,
  "has_image": false,
  "has_file": false,
  "text_excerpt": "用户文本核心内容",
  "input_intent_hint": "初步表达了一个模糊需求"
}
```

**系统提示词建议**

```text
你是需求入口分析器。
你的任务不是回答用户，而是识别本轮输入中是否包含以下信息：
1. 可直接理解的文字需求
2. 图片输入
3. 文件输入

请严格输出 JSON。
规则：
- has_text/has_image/has_file 只输出布尔值
- text_excerpt 提炼用户文字中的核心意思，没有则输出空字符串
- 不要臆造不存在的图片或文件
```

### 8.3 选择器节点：图片分流

条件：
- 如果 `has_image = true`，进入图片解析器
- 否则进入空分支

空分支建议接一个文本处理节点或变量赋值节点，输出空字符串 `""`，便于后续聚合。

### 8.4 大模型节点：图片解析器

**目的**
读取用户上传图片，提取与需求有关的信息，而不是泛化描述整张图。

**输入**
- `user_text` = 分析器输出 `text_excerpt`
- 视觉理解输入：用户图片

**输出格式**
JSON：

```json
{
  "image_requirement_info": "图片中与需求有关的关键信息",
  "image_risk_or_uncertainty": "图片信息中仍不清楚的点"
}
```

**系统提示词建议**

```text
你是需求图片解析器。
请只提取与用户需求有关的信息，例如界面结构、页面模块、视觉风格、表单字段、图中标注、业务对象。
不要做泛化审美评论。
如果图片和需求关系不明确，请在 image_risk_or_uncertainty 中指出。
严格输出 JSON。
```

### 8.5 选择器节点：文件分流

条件：
- 如果 `has_file = true`，进入文件解析器
- 否则进入空分支

### 8.6 插件节点：文件解析器

参考本地文档中的“解析文件”教程，使用链接读取类插件。

**输入**
- `url` = 开始节点上传文件变量

**输出**
- `file_content`：文件正文内容

如果同一轮可能存在多个文件，第一版建议先限制为单文件；多文件后续可用数组+循环节点扩展。

### 8.7 变量聚合节点

建议使用两个变量聚合节点：
- 聚合图片解析结果
- 聚合文件解析结果

聚合策略均为“返回第一个非空值”。

### 8.8 大模型节点：多模态整合与动作判断器

**目的**
把文字、图片、文件统一成一个需求上下文，并给出当前动作建议。

**输入**
- `user_input`
- `text_excerpt`
- `image_requirement_info`
- `file_content`

**输出格式**
JSON：

```json
{
  "unified_requirement_context": "整合后的需求上下文",
  "current_action": "clarify",
  "missing_info": [
    "目标用户不明确",
    "交付形式不明确"
  ],
  "rewrite_instruction": "",
  "save_ready": false
}
```

`current_action` 约束值：
- `clarify`
- `generate_summary`
- `rewrite_summary`
- `save_summary`

**系统提示词建议**

```text
你是需求流转分析器。
请综合用户本轮输入、图片解析结果、文件解析结果，判断当前最合适的下一步动作。

动作定义：
- clarify：需求仍模糊，需要补充关键细节
- generate_summary：信息已基本足够，应该生成需求理解摘要
- rewrite_summary：用户正在要求修改、补充、删减已有摘要
- save_summary：用户明确表达保存、入库、记录当前摘要

判断规则：
- 优先理解用户当前意图，不要机械按流程走
- 如果仍缺少目标、对象、范围、约束、交付物中的关键项，优先 clarify
- 如果用户明确说“生成摘要/总结一下/整理一下”，优先 generate_summary
- 如果用户明确说“修改、改一下、补充一下、把xxx改成xxx”，优先 rewrite_summary
- 如果用户明确说“保存、存一下、入库”，优先 save_summary

严格输出 JSON，不要附加解释。
```

### 8.9 意图识别节点：动作路由

使用完整模式，输入为 `current_action` + `user_input` + `unified_requirement_context`。

意图分类建议：
- 需求澄清
- 生成需求摘要
- 改写需求摘要
- 保存需求摘要

建议追加提示词：
- 当 `current_action` 已给出明确动作时，优先服从该动作。
- 如果文本同时包含多个动作，以用户当前最直接要求为准。

### 8.10 大模型节点：需求澄清问题生成器

**目的**
只补最关键的缺失信息，不超过 5 个问题。

**输入**
- `context` = `unified_requirement_context`
- `missing_info`

**输出格式**
JSON：

```json
{
  "need_clarify": true,
  "question_count": 3,
  "questions_text": "1. 这个需求主要给谁用？\n2. 你最希望先解决哪个问题？\n3. 你期望最终交付成什么形式？"
}
```

**系统提示词建议**

```text
你是产品需求澄清助手。
请根据当前需求上下文，只提出最必要的问题。
规则：
- 问题总数 1 到 5 个
- 优先覆盖：目标、用户对象、范围、约束、交付物
- 语气自然，不要盘问
- 如果信息已足够清晰，则 need_clarify=false，questions_text 置空
严格输出 JSON。
```

输出后接选择器：
- `need_clarify = true`：走输出节点，把问题发给用户，然后结束
- `need_clarify = false`：直接流转到摘要生成器

### 8.11 大模型节点：需求摘要生成器

**输出格式**
Markdown

**建议摘要结构**

```text
# 需求理解摘要
## 1. 需求目标
## 2. 目标用户
## 3. 核心使用场景
## 4. 功能范围
## 5. 非功能要求/约束
## 6. 待确认事项
## 7. 下一步建议
```

**系统提示词建议**

```text
你是产品经理助手，负责把模糊需求整理成“需求理解摘要”。
要求：
- 语言简洁、清晰、可执行
- 不要编造成熟需求，缺失信息要明确标记为待确认事项
- 优先总结用户已经明确表达的信息
- 输出中文 Markdown
```

建议在摘要生成后增加一个输出节点，把摘要实时返回给用户；再进入结束节点统一收尾。

### 8.12 大模型节点：需求摘要改写器

**输入**
- `latest_summary`
- `rewrite_instruction` 或 `USER_INPUT`

**输出格式**
Markdown

**系统提示词建议**

```text
你是需求摘要改写助手。
请根据用户的新要求，对最近一次需求理解摘要进行改写。
规则：
- 保留未被要求修改的有效内容
- 只修改用户明确提到的部分
- 如果用户补充了新信息，要合并到对应章节
- 输出完整的新版本摘要，不要只输出改动点
```

### 8.13 新增数据节点：保存需求摘要

写入表：`requirement_summaries`

建议字段：
- `conversation_name`，String
- `raw_user_input`，String
- `summary_markdown`，String / Long Text
- `source_context`，String / Long Text
- `action_type`，String
- `created_at`，String

字段赋值建议：
- `conversation_name` = `CONVERSATION_NAME`
- `raw_user_input` = `USER_INPUT`
- `summary_markdown` = 最近一次摘要生成器或改写器输出
- `source_context` = `unified_requirement_context`
- `action_type` = `save_summary`
- `created_at` = 固定时间字符串或由上游生成

保存成功后，可接一个输出节点：

```text
已保存当前版本的需求理解摘要。你可以继续补充信息，我会在此基础上继续更新。
```

## 9. 数据库设计

### 9.1 数据表

表名：`requirement_summaries`

| 字段名               | 类型     | 必填  | 说明         |
| ----------------- | ------ | --- | ---------- |
| conversation_name | String | 是   | 会话名称       |
| raw_user_input    | String | 否   | 本轮用户原始输入   |
| summary_markdown  | String | 是   | 最新需求摘要     |
| source_context    | String | 否   | 多模态整合后的上下文 |
| action_type       | String | 是   | 保存动作类型     |
| created_at        | String | 是   | 保存时间       |

### 9.2 是否需要多用户模式

如果该应用会被多个真实用户同时使用，建议创建数据库时启用多用户模式，避免不同用户的数据混淆。

## 10. 推荐对话输出策略

- 中间状态用输出节点提示：
  - “我先帮你整理一下现有信息。”
  - “我正在解析你提供的文件内容。”
  - “我已经理解大致方向，还差几个关键点想确认。”
- 这类中间提示建议设置为“不写入会话历史”，避免污染模型后续上下文。
- 真正的澄清问题和需求摘要应写入会话历史。

## 11. 异常与边界处理

- 图片解析失败：返回“图片已收到，但暂时无法完整解析，请补充一句这张图想表达什么。”
- 文件解析失败：返回“文件已收到，但暂时无法读取正文，请补充文件重点或重新上传。”
- 用户一句话直接要求保存，但此前无摘要：先走摘要生成，再保存。
- 用户要求改写，但会话里没有上一版摘要：先生成首版摘要，再提示可继续修改。
- 用户同时发来图片和文件：都解析，再统一整合。
- 用户只说“帮我整理一下”：默认生成摘要。

## 12. 调试用例

### 用例 1：只有模糊文字

输入：
“我想做一个给销售团队用的客户跟进助手，大概能提醒事情、记录沟通。”

预期：
- 动作判断为 `clarify`
- 输出 3-5 个问题

### 用例 2：文字足够明确

输入：
“帮我整理一个需求摘要：做一个内部报销审批系统，员工提交报销单，主管审批，财务打款，需要保留审批记录。”

预期：
- 动作判断为 `generate_summary`
- 直接输出结构化摘要

### 用例 3：要求改写

前文已有摘要，本轮输入：
“把目标用户从销售改成渠道运营，再补充一个数据看板模块。”

预期：
- 动作判断为 `rewrite_summary`
- 输出完整新摘要

### 用例 4：要求保存

前文已有最近摘要，本轮输入：
“可以，先保存这一版。”

预期：
- 动作判断为 `save_summary`
- 数据库新增一条记录
- 返回保存成功提示

### 用例 5：文件辅助说明

输入：
上传 PDF 需求草稿，并说“结合这个文档，帮我继续澄清需求。”

预期：

- 文件内容被整合
- 输出高价值澄清问题

## 13. 实施建议

### 方案 A：单对话流直接实现

适合第一版快速上线，节点少，便于调试。

### 方案 B：图片解析、文件解析封装为子工作流

适合后续扩展，便于复用和维护。

推荐先做方案 A，跑通主路径后再拆子工作流。

## 14. 待确认问题

1. 用户上传的“文件”在你的目标渠道里是否允许直接通过对话流拿到文件对象。
2. 图片解析是否必须做 OCR 级别提取，还是只需产品理解层描述。
3. 保存数据库时，是否需要记录用户 ID、渠道 ID、摘要版本号。
4. 这条流程最终是发布到扣子商店、网页 AI 对话组件，还是 API&SDK。

## 15. 最终建议

这条需求最适合用“对话流 + 大模型节点 + 意图识别节点 + 选择器节点 + 文件解析插件 + 新增数据节点”的组合实现。

第一版建议把重点放在三件事：
- 多模态输入先稳定解析
- 动作判断只保留 4 类
- 澄清问题控制在 3-5 个

这样可以先把“模糊需求变成可理解摘要”的主链路跑通，再逐步增加摘要版本管理、历史查询、结构化表单补充等高级能力。
