# PRD: AI 产品工作台 MVP 复刻需求

## 1. 背景与目标

目标是在一个新的账号和全新仓库里，从零到一复刻当前 AI 产品工作台的 MVP 能力，但只保留最小可闭环范围，不复刻所有二阶段 Agent 化能力。

这次复刻的核心不是“完全拷贝现有仓库”，而是做出一个可独立运行、可继续迭代、能证明核心价值的 MVP：

- 保留对话式需求输入体验
- 能从需求对话生成 PRD、HTML 原型、Story、测试用例
- 能引用历史对话内容继续生成
- 支持图片粘贴、上传附件参与生成
- 支持在右侧展开查看、编辑、预览产物
- 支持导出到本地或写入飞书
- 支持本地 Codex 代理和通用大模型 API 两种调用方式

一句话定义：

做一个本地优先、浏览器使用的 AI 产品工作台 MVP，让用户能从一段需求对话快速产出并管理 PRD、HTML 原型、Story、测试用例和评审结果。

## 2. 产品目标

- 让用户从一句需求出发，在一个页面内完成“对话 -> 产物生成 -> 编辑预览 -> 导出/写入”的最小闭环
- 把对话内容沉淀为结构化产物，而不是只停留在聊天记录里
- 让产物能被继续引用、编辑和评审，形成项目级资产
- 在不强依赖云端账号体系的前提下，支持本地运行和本地持久化
- 在模型侧同时支持 `local_proxy` 和 `openai_api`

## 3. 核心用户

- 独立产品经理
- 创业团队中的产品负责人
- 需要快速产出 PRD、原型、研发拆解和测试材料的 AI 产品从业者

## 4. MVP 范围

### 4.1 P0 必做

1. 工作台基础壳层
- 左侧项目/会话区
- 中央对话区
- 底部或中部输入区
- 右侧产物面板

2. 项目与会话
- 新建项目
- 项目内持续对话
- 历史消息持久化
- 历史 AI 回复可选中并作为后续生成引用上下文

3. 输入能力
- 文本输入
- 粘贴图片
- 上传图片
- 上传普通附件
- 已上传附件可在当前轮对话中展示

4. 任务生成
- 普通对话
- 生成 PRD
- 生成 HTML 原型
- 生成 Story
- 生成测试用例

5. 产物管理
- 每次生成都落为独立 artifact
- 右侧面板可展开查看当前 artifact
- 支持编辑 artifact
- PRD 支持文本预览
- HTML 支持 iframe 预览
- Story / 测试用例支持表格预览

6. 导出与交付
- PRD 可保存为本地 Markdown
- HTML 可保存为本地 HTML 文件
- PRD 可写入飞书文档
- Story 和测试用例可写入飞书多维表
- Story 和测试用例也可导出本地 JSON / CSV / Markdown 三选一，至少实现一种表格友好格式

7. 评审能力
- 支持对 PRD 发起评审
- 支持对测试用例发起评审
- 评审结果也作为独立 artifact 保存

8. Provider 切换
- 支持 `local_proxy` 模式
- 支持 `openai_api` 模式
- 前端可配置当前 provider

9. 本地持久化
- 项目
- 会话
- 消息
- 附件元数据
- 产物
- 发布/导出结果

### 4.2 P1 应该做

- 产物间引用关系，例如“基于当前 PRD 生成测试用例”
- 自动带入已选引用消息和已选产物摘要
- PRD、Story、测试用例的结构化模板校验
- 导出后在 UI 中显式展示本地路径或飞书链接
- 支持重新生成同类型产物并保留历史版本

### 4.3 明确不做

- 用户登录、团队协作、权限系统
- 在线多人编辑
- 真正的画布式原型编辑器
- 飞书 OAuth 登录
- 自动执行浏览器/终端 Agent 工作流
- 复杂调度、自动化、Run/Step 编排系统

## 5. 关键术语

- `project`：一个需求主题下的工作空间
- `conversation`：项目内的一段会话
- `message`：聊天消息
- `attachment`：图片或普通附件
- `artifact`：每次生成的结构化产物
- `provider`：模型调用方式，取值为 `local_proxy` 或 `openai_api`

说明：

当前老仓库中的 `dev_tasks` 可以在本次 MVP 中命名为 `story`。如果实现时为了兼容老逻辑继续使用 `dev_tasks` 作为内部字段，前端文案必须统一展示为 `Story` 或 `研发 Story`。

## 6. 用户故事

### US-001: 新建项目并开始需求对话

**描述：** 作为产品经理，我希望输入一句需求就能开始项目对话，这样我可以把想法沉淀到项目内持续迭代。

**验收标准：**
- [ ] 用户可以创建项目
- [ ] 创建项目后自动进入该项目会话页
- [ ] 用户发送第一条消息后，消息会持久化
- [ ] 刷新页面后仍能看到该项目和消息历史

### US-002: 引用历史 AI 回复继续生成

**描述：** 作为用户，我希望勾选历史 AI 回复作为引用上下文，这样后续产物会更贴合之前的讨论。

**验收标准：**
- [ ] AI 回复消息支持“引用为上下文”
- [ ] 可同时选择多条回复
- [ ] 发起下一轮生成时会带上所选引用
- [ ] UI 能显示当前已引用条数
- [ ] 取消引用后不再带入

### US-003: 粘贴图片和上传附件

**描述：** 作为用户，我希望直接粘贴截图或上传附件，这样可以把竞品图、草图和补充材料带入生成过程。

**验收标准：**
- [ ] 输入框支持粘贴图片
- [ ] 支持选择本地文件上传
- [ ] 图片与普通文件均可上传
- [ ] 已上传附件在输入区可见
- [ ] 图片附件能作为多模态输入参与生成
- [ ] 非图片附件至少以文件名形式参与上下文提示

### US-004: 从需求对话生成 PRD

**描述：** 作为用户，我希望基于当前需求对话直接生成 PRD，这样我可以快速进入需求梳理阶段。

**验收标准：**
- [ ] 用户可点击“生成 PRD”
- [ ] 系统生成结构化 PRD artifact
- [ ] PRD 至少包含：背景与目标、用户与场景、核心功能、交互/流程、验收标准
- [ ] PRD 在右侧面板中可查看、编辑、预览
- [ ] PRD 生成后自动绑定到当前项目

### US-005: 从需求对话生成 HTML 原型

**描述：** 作为用户，我希望根据需求描述直接生成可预览的 HTML 原型，这样我能快速验证页面方向。

**验收标准：**
- [ ] 用户可点击“生成 HTML 原型”
- [ ] 系统返回单文件 HTML artifact
- [ ] 右侧面板支持 iframe 预览
- [ ] 用户可编辑 HTML 源码
- [ ] 编辑后可立即重新预览效果

### US-006: 生成 Story

**描述：** 作为用户，我希望把需求拆成 Story 列表，这样我可以继续推进研发排期或导入协作工具。

**验收标准：**
- [ ] 用户可点击“生成 Story”
- [ ] Story 以表格结构保存为 artifact
- [ ] 表格至少包含：标题、模块、描述、依赖、验收标准、优先级、状态
- [ ] Story 可在右侧表格预览
- [ ] Story 支持编辑后保存

### US-007: 生成测试用例

**描述：** 作为测试或产品角色，我希望基于需求或 PRD 生成测试用例，这样我可以快速得到可执行的验证清单。

**验收标准：**
- [ ] 用户可点击“生成测试用例”
- [ ] 测试用例以表格结构保存为 artifact
- [ ] 表格至少包含：用例名称、前置条件、测试步骤、预期结果、关联功能点、优先级、状态
- [ ] 测试用例可在右侧表格预览
- [ ] 测试用例支持编辑后保存

### US-008: 在右侧面板查看、编辑和预览产物

**描述：** 作为用户，我希望在不离开对话页的情况下查看和修改产物，这样我能保持连续工作流。

**验收标准：**
- [ ] 点击消息中的 artifact 卡片可打开右侧面板
- [ ] 右侧面板可切换查看不同 artifact
- [ ] PRD 显示 Markdown 阅读态
- [ ] HTML 显示预览态
- [ ] Story / 测试用例显示表格态
- [ ] 所有 artifact 至少支持一种编辑方式

### US-009: 导出到本地或写入飞书

**描述：** 作为用户，我希望把产物导出到本地或写入飞书，这样可以继续交付给团队使用。

**验收标准：**
- [ ] PRD 支持导出 Markdown 到本地
- [ ] HTML 支持导出 HTML 到本地
- [ ] PRD 支持写入飞书文档
- [ ] Story 支持写入飞书多维表
- [ ] 测试用例支持写入飞书多维表
- [ ] 成功后 UI 中能看到本地路径或外部链接
- [ ] 失败时 UI 中能看到明确错误信息

### US-010: 发起 PRD 和测试用例评审

**描述：** 作为用户，我希望对已生成的 PRD 和测试用例发起评审，这样我可以快速发现范围、逻辑和验证上的缺口。

**验收标准：**
- [ ] 用户可对 PRD 发起评审
- [ ] 用户可对测试用例发起评审
- [ ] 评审结果保存为独立 artifact
- [ ] 评审结果至少包含：结论、问题点、建议、下一步动作
- [ ] 评审时默认引用被评审 artifact 内容

### US-011: 在 provider 之间切换

**描述：** 作为用户，我希望根据场景切换本地代理或 API，这样我可以平衡成本、稳定性和速度。

**验收标准：**
- [ ] 设置区可切换 `local_proxy` 与 `openai_api`
- [ ] 当前 provider 状态在 UI 中可见
- [ ] 当前 provider 会影响实际生成请求
- [ ] provider 调用失败时，错误提示可区分是服务不可达、鉴权失败还是返回格式异常

## 7. 功能需求

### 7.1 工作台布局

- FR-1: 页面必须包含左侧项目/会话区、中央对话区、右侧产物区
- FR-2: 用户在中央区域完成输入、查看消息和发起生成动作
- FR-3: 点击 artifact 后，右侧面板展开显示对应内容

### 7.2 对话与引用

- FR-4: 系统必须支持项目内多轮对话
- FR-5: 系统必须持久化消息历史
- FR-6: AI 回复必须支持被标记为“引用上下文”
- FR-7: 发起生成时，系统必须带入当前输入、已引用消息、可用项目摘要和当前附件

### 7.3 附件输入

- FR-8: 系统必须支持文件上传
- FR-9: 系统必须支持在输入框粘贴图片
- FR-10: 图片附件应在支持的 provider 下作为图片输入发送
- FR-11: 非图片附件在 MVP 中可不做内容解析，但必须被记录并展示

### 7.4 任务类型

- FR-12: 系统必须支持 `chat`、`prd`、`html_prototype`、`story`、`test_cases`
- FR-13: 每种任务类型都必须生成独立 artifact，而不是只输出在聊天消息里
- FR-14: Artifact 必须记录类型、标题、内容、状态、创建时间、更新时间

### 7.5 Artifact 呈现

- FR-15: PRD 使用 Markdown 存储和预览
- FR-16: HTML 原型使用完整 HTML 文本存储，并支持 iframe 预览
- FR-17: Story 与测试用例使用结构化 rows 存储和表格预览
- FR-18: Artifact 支持编辑后保存

### 7.6 导出与飞书集成

- FR-19: PRD 支持导出本地 Markdown 文件
- FR-20: HTML 原型支持导出本地 HTML 文件
- FR-21: PRD 支持写入飞书 Doc
- FR-22: Story 与测试用例支持写入飞书 Bitable
- FR-23: 导出和发布结果必须回写到 artifact 上

### 7.7 评审

- FR-24: 系统必须支持基于 PRD 生成 PRD Review artifact
- FR-25: 系统必须支持基于测试用例生成 Test Case Review artifact
- FR-26: Review artifact 必须可阅读、可复制、可继续追问

### 7.8 Provider 与配置

- FR-27: 系统必须支持 `local_proxy`
- FR-28: 系统必须支持 `openai_api`
- FR-29: 系统必须通过环境变量或设置面板配置 provider 相关参数

## 8. 非功能要求

- NFR-1: 项目默认本地运行，浏览器访问
- NFR-2: 数据默认本地持久化，可使用 SQLite + 文件系统
- NFR-3: 接口报错必须转为用户可理解的中文提示
- NFR-4: HTML 预览在桌面端能稳定打开
- NFR-5: 基础流程在 mock 或真实 provider 下至少有一层自动化测试
- NFR-6: 上传、导出、发布失败时不能只显示 `Failed to fetch`

## 10. 建议的技术方案

### 10.1 技术栈建议

- 前端：React + Vite
- 后端：Node.js 单体服务
- 存储：SQLite + 本地文件
- 模型接入：OpenAI 兼容接口 + 本地 Codex 代理
- 导出：本地文件系统
- 飞书：应用身份直写 Doc / Bitable

### 10.2 实现原则

- 先做单体项目，避免过早拆前后端服务
- 先把 artifact 模型和 UI 面板做对，再扩功能
- 先做结构化 JSON 输出，再映射为 PRD/表格/HTML 视图
- Review 先做“生成评审结论 artifact”，不要上复杂审批流

## 15. 给 Codex 的执行要求

如果把这份文档直接交给 Codex，请额外要求它遵守下面几点：

1. 先做 MVP，不要引入多用户、复杂权限、Agent 编排、自动调度等超纲能力。
2. 每种任务都必须生成独立 artifact，并能在右侧面板打开。
3. Story 可以复用现有 `dev_tasks` 的结构，但对外文案必须统一成 `Story`。
4. 先保证本地闭环，再接飞书和真实 provider。
5. 所有“生成成功”都必须有真实可见结果，不能只在聊天里回复一段文本。
6. 导出或飞书写入成功后，前端必须显示本地路径或外部链接。
7. 报错信息必须对用户友好，避免只显示底层报错或 `Failed to fetch`。

MVP 必须包含：项目/会话、对话输入、历史引用、图片粘贴与附件上传、PRD/HTML/Story/测试用例生成、右侧 artifact 展开查看、编辑与预览、本地导出、飞书写入、PRD 和测试用例评审、`local_proxy` 与 `openai_api` 切换。  
请严格按 MVP 范围实现，不要扩展到多用户、复杂 Agent 执行编排、自动化调度或在线协作。  
每完成一个阶段后都要补最小验证：单测、必要的前端构建验证、至少一条端到端主链路验证。
